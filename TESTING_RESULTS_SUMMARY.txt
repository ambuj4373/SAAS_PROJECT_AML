🔍 KEY LEARNINGS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. INPI API IS SENSITIVE TO TIMEOUTS
   └─ Large companies with many directors take longer to fetch
   └─ Need exponential backoff + longer timeout tolerance

2. SIREN VALIDATION IS CRITICAL
   └─ API can return valid but different company data
   └─ Must validate SIREN matches, not just accept any response

3. FINANCIAL DATA IS NOT RELIABLE FROM INPI ALONE
   └─ API returns empty financial records for most companies
   └─ Need fallback to public sources (INSEE, BNP Paribas, etc)

4. DIRECTOR DATA EXTRACTION REQUIRES MULTI-PATH APPROACH
   └─ Some companies don't expose directors via standard path
   └─ Need alternative field checking + formality record parsing

5. API RELIABILITY REQUIRES RESILIENCE LAYER
   └─ Raw requests.get() insufficient for production
   └─ Need caching, retries, circuit breaker, metrics

6. DATA QUALITY TRANSPARENCY IS ESSENTIAL
   └─ Risk scores without quality flags are misleading
   └─ Must track: source, completeness, confidence levels
